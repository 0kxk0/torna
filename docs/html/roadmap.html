Roadmap

What's shipped, what's next, and where the line is between MVP-ready and mainnet-ready.

Shipped (v3)

Core engine

  • Full B+ tree on Solana: insert, find, range scan, delete.
  • Multi-account span with PDA-derived addressing — no local state.
  • Splits during insert; cascading rebalance during delete; root collapse on merge.
  • Leaf chain for ordered iteration across many accounts.

Parallel-safe fast paths

  • InsertFast, DeleteFast, BulkInsertFast, BulkDeleteFast.
  • Header read-only, only target leaf writable. Disjoint writes parallelize.
  • Devnet benchmark: 6 concurrent inserts in one slot, 6.47× speedup.

Operational primitives

  • Authority model with TransferAuthority instruction.
  • CPI to system_program for on-demand node allocation — only consumed accounts pay rent.
  • Source-available license, public repo, judge-ready for hackathons.

Tooling

  • TypeScript client with all 12 instruction builders.
  • Rust SDK crate (torna-sdk) with CPI helpers and PDA derivation.
  • Example consumer program (mini orderbook) using CPI.
  • Anchor-compatible IDL.
  • Local validator CI script — zero-cost iteration.
  • Invariant checker with nine on-chain invariants validated post-operation.
  • Seeded random stress test.

v3.1 — runtime variable value sizes — shipped

Status: landed 2026-05-12

The on-chain stride math is now parameterised by th->value_size across every leaf helper, ix-data parser, and return-data writer. Stack arrays remain sized at the compile-time bound VAL_SIZE_MAX = 64. InitTree takes value_size in ix_data and validates it falls within [1, 64].

Before v3.1: value_size was 32 bytes at compile time. Fine for many workloads — a Pubkey fits, a composite struct of (u64, u64, status) packs.

After v3.1: each tree picks its own value_size at creation. An NFT marketplace can use 64-byte listings; a DAO can use 40-byte vote records; an oracle index can use 8-byte u64 prices. All within the same deployed program; no recompilation needed.

Keys remain fixed at 32 bytes — they encode composite identifiers well enough that runtime key_size hasn't been a constraint. Variable key_size is on the longer-term list but not gated.

v3.2 — multi-authority delegates — shipped

Status: landed 2026-05-12

An optional side account at PDA ("torna_dlg", tree_id) holds up to eight additional signers per tree. Primary authority manages the list via IX_ADD_DELEGATE and IX_REMOVE_DELEGATE; authority transfer remains primary-only by design.

Every write instruction now goes through a single authority check that accepts either the primary signer or any registered delegate. For tx's where a delegate is signing, the caller appends the delegate side account to the instruction's account list; the program scans for it, verifies its PDA, and validates the signing relationship. End-to-end verified on devnet — delegate-signed FAST insert runs in ~4,900 CU.

v3.3 — Anchor codegen smoke test

The IDL exists. v3.3 runs it through Anchor's TypeScript codegen and validates that the generated client can actually call every instruction end-to-end without manual surgery. Cheap work, removes friction for Anchor-native teams.

v3.4 — on-chain events — shipped

Status: landed 2026-05-12

The program emits five structured event types via sol_log_data: Insert, Delete, AuthorityChange, DelegateAdded, DelegateRemoved. Each event is one log line with a 1-byte discriminator and a packed payload. The TypeScript client ships a decoder (events.ts) that turns tx logs into a typed TornaEvent union.

Indexers, Geyser plugins, and analytics services can subscribe to program logs and rebuild tree state externally without polling on-chain accounts. The format is intentionally minimal — one discriminator byte, fixed-shape payload — so downstream consumers don't pay decoding overhead per event.

Insert and Delete events carry the key, value, and target leaf index. AuthorityChange and the two delegate events carry the relevant pubkeys. The full set is emitted across the existing instructions (no new instruction surface), so existing clients keep working unchanged.

v3.5 — atomic multi-leaf bulk insert

Today's BulkInsertFast targets one leaf. v3.5 supports inserting into multiple leaves in a single transaction — useful for batch order placement that hits several price levels at once, or batched vote casting that touches different time-bucketed leaves.

v4 — variable fanout

Fanout is currently 64 at compile time. Different workloads want different fanouts: high-cardinality keyspaces want bigger nodes (fewer levels, more concurrent leaves); low-cardinality dense keys want smaller nodes (less wasted space per leaf). v4 makes this a tree-level parameter set at InitTree, with the same 8 KB node size budget.

v4.x — formal verification

After the API stabilizes through v3.x, run the C program through a formal verifier (CBMC, Frama-C, or KLEE-style symbolic execution) to prove the B+ invariants hold under all reachable states. Pairs with a structured fuzzer that exercises insert/delete sequences and asserts invariants after every operation.

This is the gate for an audit-ready mainnet release. Investing in it before audit reduces audit cost and dramatically narrows the bug surface.

Path to mainnet

Reading the roadmap in order: v3.1 → v3.2 give the SDK shape consumers need. v3.3 → v3.5 polish integration ergonomics and observability. v4 unlocks tuning per use case. v4.x is the rigor pass.

Realistic timeline (single-engineer focus): six months from v3 ship to mainnet-ready, including a third-party audit. Faster with a small engineering team. The architecture itself is sound; the remaining work is hardening and ergonomics, not invention.

Things explicitly not planned

Some things stay out of scope to keep the primitive sharp:

  • Built-in matching engine. Torna is the storage. Order matching belongs in the consumer program that uses Torna.
  • Token transfers. Same — Torna stores keys and values, not assets.
  • Cross-chain. Solana-only. Other chains can have their own implementations.
  • Off-chain components. No mandatory indexer, no required RPC service. The tree is fully on-chain; clients are stateless beyond the program ID and tree ID.

Discipline about what Torna is not is part of what keeps it useful as a primitive.
