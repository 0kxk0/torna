The parallelism benchmark

The headline experiment. We built a tree with 200 sparse keys (stride-1000 to create six leaves with non-overlapping ranges). Then we issued six insert transactions, each targeting a different leaf. Two ways:

  • Serial: send one transaction, wait for confirmation, send the next, wait, send the next, and so on for all six.
  • Parallel: send all six transactions simultaneously via Promise.all(), with the same recent blockhash, signed in advance.

Every parallel transaction landed in the same slot. Devnet was not under unusual load. Same payer, same RPC, same program. The only difference between the runs was whether each transaction declared the tree header as writable (serial path) or read-only (parallel FAST path).

This is the result you'd expect, given the Solana scheduler's rules. It's also the result Phoenix and OpenBook cannot produce, because their data structure makes header read-only impossible.

Cost economics

Each B+ tree node is an 8 KB Solana account. Rent-exempt: ~0.057 SOL per node. The tree header is 80 bytes: ~0.0008 SOL.

A 200-entry tree built with sparse keys typically settles at five nodes — one internal root plus four leaves. Total:

ItemCost (SOL)
Tree header (80 B)~0.0008
5 node accounts (8 KB each)~0.143
Transaction fees (200 inserts × ~5000 lamports)~0.001
Total~0.18 SOL

If we'd done worst-case pre-allocation (the naïve approach where every insert reserves up to height+1 spare accounts upfront), the same workload would have cost ~17 SOL — most of it wasted on accounts that were never consumed. The PDA-based on-demand allocation eliminates that waste entirely.

The cost looks steep at the high end — that's an inevitable consequence of storing 8 KB per account with Solana's rent model. The relevant comparison isn't to free; it's to alternatives. A Phoenix slab capped at 10 MB costs ~57 SOL (one rent-exempt 10 MB account) and holds far fewer entries than a height-3 Torna tree at the same SOL budget, with the added constraint of zero parallelism. spl-account-compression is dramatically cheaper at scale but offers no order.

For real workloads, the cost lives in the height-2 to height-3 range. A DEX orderbook with 100k open orders fits comfortably in a height-3 tree built with sparse keys — the actual cost depends on how the keyspace fills.

Reproducing these

From the repo:

# build + deploy
make
solana program deploy out/torna_btree.so --program-id out/torna_btree-keypair.json

# the parallel benchmark
cd client && npm install
SETUP_N=200 npx tsx src/bench_parallel.ts

The script outputs every transaction signature plus the slot it landed in. The "all 6 in the same slot" claim is verifiable on Solana Explorer for any reader.

For a fully local run with no devnet costs, use ./scripts/ci.sh which spins up solana-test-validator, deploys, runs every demo and invariant check, then tears down.
