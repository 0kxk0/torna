Torna

A B+ tree on Solana. Written in C, spread across many accounts, and built so that writes to disjoint regions run in the same slot. It's the database primitive Solana programs reach for when they need ordered keys, range scans, and write throughput a single 10 MB account can't give them.

What this actually is

Torna is a small Solana program that owns nothing except a logical B+ tree spread across many accounts. Other programs CPI into it the way a backend service uses Postgres. The interface is what you'd expect from a sorted store: insert, find, delete, range scan, plus the parallel-safe fast paths for high-throughput workloads.

The closest analog from outside crypto is SQLite — a single ordered table, embedded, no daemon. Torna isn't trying to be a full DBMS. It's the one missing primitive: ordered keys, on-chain, written in a way that doesn't fight Solana's parallelism model.

Why it matters

Solana's account model is a hash table. Pubkey to bytes. Beautiful for concurrency — transactions declare their accounts and the scheduler runs disjoint transactions in parallel. Bad for anything that needs order: an orderbook, a price-sorted NFT marketplace, a leaderboard, a time-series oracle.

Today, every team that needs sorted state on Solana writes their own slab allocator inside a single 10 MB account. That gets them order, fast. It also caps them at one writer per slot, because every order placement contends on the same master account. Phoenix and OpenBook are both built this way. The result is a hard ceiling on throughput that has nothing to do with Solana's actual capacity.

Torna takes the same data structure (a balanced ordered tree) and spreads it across accounts. Two orders at different price levels write to different leaves. The Solana scheduler sees disjoint write sets. They run together.

Where it stands today

The program is deployed on devnet at C4qNsrX92Nn5z8HwSZCcioMgRoLG6BzCCK6X7kRjEFoD. Twelve instructions, including a parallel-safe fast path for both insert and delete. A Rust SDK with CPI helpers and a TypeScript client are in the repo, along with an example consumer program (a tiny orderbook) that uses Torna over CPI.

What's not there yet: a mainnet audit, runtime-variable key and value sizes (today they're fixed at 32 bytes each), multi-authority delegation, and a few smaller items on the roadmap. The architecture is in place. The hardening pass is what's left.
