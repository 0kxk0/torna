License & Contact

Torna is source-available — public for evaluation, education, and hackathon judging. Production deployment, commercial use, and derivative works require a written license.

The short version

What you can do
  • Read the source code, in full, on GitHub.
  • Build it locally, run it on your own machine, deploy to a personal devnet wallet for evaluation.
  • Submit the unmodified project to recognized Solana hackathons or grant programs.
  • Cite, reference, or analyze the design in academic or technical writing.
What you can't do (without a written license)
  • Deploy the program — or any program derived from it — to Solana mainnet, the public devnet beyond evaluation, or any SVM-compatible chain.
  • Use any part of the source code, compiled binaries, or design patterns in a commercial product, service, or revenue-generating activity.
  • Create derivative works, ports, adaptations, or reimplementations of the algorithms and on-chain layouts.
  • Sublicense, sell, lease, or transfer the software.
  • Build a competing on-chain database primitive using techniques documented or implemented in this repository.

The full license text is in the LICENSE file on the GitHub repository.

Why this license

Two things at the same time. The code needs to be public so judges, peers, and prospective integrators can evaluate it, run the benchmarks, and verify the design claims. Public source is the price of credibility for an infrastructure project at this stage.

At the same time, the value of an infrastructure primitive is in being the standard implementation that integrators pay to embed. Permissive licensing at this stage would let anyone fork, deploy, and capture the market position without contributing back. The strict license isn't anti-community — it's pro-sustainability of the project itself.

Once the project reaches a maturity threshold (typically post-audit, post-mainnet, post-broad-adoption), the license can be relaxed. Many infrastructure projects follow this trajectory — Redis, MongoDB, Elastic — moving toward more permissive terms once revenue and ecosystem position are established.

Commercial licensing

If you want to use Torna in production, embed it in a commercial product, or build a service on top of it, we're set up to make that work. The license process is intentionally lightweight for legitimate integrators:

  1. Describe the use case, the deployment scope (mainnet/private chain/SaaS/etc.), and your organization.
  2. We'll come back with terms — typically a flat fee plus optional revenue share for high-volume deployments, or a custom arrangement for strategic partnerships.
  3. Sign, deploy, ship.

For early-stage protocols and well-aligned teams, the terms can be very favorable. The intent is to make Torna easy to adopt for teams that are building real products, while preserving the project's commercial position against extractive forks.
